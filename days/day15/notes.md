## シナリオ1: ユーザーから「画面が真っ白」と報告

### 切り分け手順
```
Step 1: サービスの状態確認（最大粒度）
  aws ecs describe-services → running 数を確認
  running=0 なら「タスクが死んでいる」→ コード関係なし

Step 2: ログ確認
  aws logs tail でエラーを探す
  エラーなし → アプリは正常動作 → フロントエンドの問題かも

Step 3: 個別の切り分け
  curl でAPIを直接叩く
  ブラウザDevTools（Network/Console）を確認
```

### ポイント
「大きいところから小さいところへ」順番に切り分ける。
いきなりコードやブラウザを見るのは効率が悪い。

### `Unexpected token '<'` が出るパターン
- 前回セッションの504エラーと同じ
- 「返ってくるはずの形式と違うもの（多くはHTML）が返ってきた」サイン
- 原因候補:
  - ブラウザキャッシュに古いJSファイル名が残っている
  - ECSデプロイ中で新旧タスクが混在
  - ALB/プロキシが502/504のHTMLを返している

## シナリオ2: SQSキューに大量のメッセージが溜まっている

### 切り分け手順
```
Step 1: Lambdaのログを確認
  aws logs tail で Lambdaが起動しているか
  → ログなし → Lambda起動していない
  → エラー → 処理失敗でリトライ中
  → 正常処理 → 単純に追いついていない
```

### Lambdaの並列化
- SQSイベントソースマッピングの設定で制御
- `BatchSize`: 1回の起動で処理する件数
- `MaximumConcurrency`: 同時実行数の上限
- デフォルトで自動スケールするが、アカウント全体で1000の上限あり

## シナリオ3: 突然S3へのアップロードが失敗

### AccessDeniedの原因候補
```
1. IAMロールのポリシーが変更された
   → dandori-infraのPR履歴を確認

2. S3バケット側のバケットポリシーで拒否された
   → バケット側で明示的にDenyが追加された可能性

3. KMS暗号化の問題
   → SSEでKMS鍵が必要なバケットで、その鍵の使用権限がない

4. SSMのS3_BUCKETが変更されて別のバケットを指している
   → そのバケットへの権限がない
   → 前回セッションで起きたパターン
```

### 切り分け手順
```
1. エラーメッセージの詳細を確認
   "arn:aws:sts::...:assumed-role/..." → どのロールが
   "resource: arn:aws:s3:::..." → どのバケットへ

2. ロールのポリシーを確認
   aws iam get-role-policy で現在の権限を見る

3. Terraformの変更履歴を確認
   git log dandori-infra/environments/production/app/ecs.tf

4. SSMの値を確認
   aws ssm get-parameter --name "/dandori-.../S3_BUCKET"
```

## 総合演習のポイント

### 障害対応の原則
```
1. 大きいところから切り分ける
   サーバーの状態 → ログ → 個別の挙動 の順

2. エラーメッセージを正確に読む
   AccessDenied なら誰が何にアクセスしようとしたかを確認
   Unexpected token なら何が返ってきたかを確認

3. 過去の事例を参照する
   前回のセッションで起きた問題は、同じパターンで再発することが多い
```

### 前回セッションとの対応
| 今回のシナリオ | 前回セッションでの体験 |
|---|---|
| 画面が真っ白 | 502/504エラー、JSONパースエラー |
| SQSメッセージ滞留 | バッチ処理で17件目で止まった件 |
| S3アップロード失敗 | SSMのS3_BUCKET設定ミス |
