## セクション1: RDSとは

### マネージドDBサービス
```
自分でDBサーバーを立てる場合:
  EC2にPostgreSQLをインストール
  → OS更新、バックアップ、冗長化、監視を全部自分で管理

RDSを使う場合:
  「PostgreSQL 17をください」と指定するだけ
  → 自動バックアップ、パッチ適用、フェイルオーバーをAWSがやってくれる
```

### DandoriAIのRDS構成
```
dandori-poc-postgres:
  → POC環境用。全テナント共有
  → db.t4g.micro、ストレージ20GB、PostgreSQL 17.6

dandori-production-postgres:
  → 本番環境用
  → db.t4g.micro、ストレージ10GB、PostgreSQL 17.6
```

### 確認コマンド
```bash
aws rds describe-db-instances \
  --query "DBInstances[*].{Name:DBInstanceIdentifier,Engine:Engine,Version:EngineVersion,Class:DBInstanceClass}" \
  --output table
```

## セクション2: マルチテナントのDB設計パターン

### 3つのパターン
```
パターン1: DBインスタンスを分ける
  テナントA → DB-A、テナントB → DB-B
  メリット: 完全分離。障害が波及しない
  デメリット: コスト高い

パターン2: スキーマを分ける ← DandoriAIのPOC
  テナントA → DB.schema-A、テナントB → DB.schema-B
  メリット: 1つのDBでコスト抑えつつ論理的に分離
  デメリット: 重いクエリが他テナントに影響する可能性

パターン3: 行レベルで分ける ← DandoriAIのProduction
  WHERE company_id = 'A' でフィルタ
  メリット: 最もシンプル
  デメリット: WHERE句を忘れるとデータ漏洩
```

### DandoriAI POCのスキーマ分離
```
dandori-poc-postgres（1つのDBインスタンス）
├── スキーマ: nhk-ep       ← nhk-epのテーブル全部
├── スキーマ: odakyu-ag    ← odakyu-agのテーブル全部
└── スキーマ: demo         ← demoのテーブル全部
```

### スキーマの接続方法
```
DATABASE_URL="postgresql://...?schema=odakyu-ag"
                                     ^^^^^^^^^^
                                     ここでスキーマを指定
→ Prismaが自動的にそのスキーマのテーブルだけを操作
```

### 実体験
- インフラエンジニアから「スキーマのオーナーを変更しましたか？」と聞かれた
- スキーマのオーナーがずれるとテーブル操作の権限に問題が起きる

## セクション3: バックアップ

### 2種類のバックアップ
```
自動バックアップ:
  → RDSが毎日自動でスナップショットを取る
  → 保持期間を設定（例: 7日間）
  → ポイントインタイムリカバリ: 「3日前の14時の状態に戻す」が可能

手動スナップショット:
  → 好きなタイミングで取る
  → 自動削除されない（自分で消すまで残る）
  → マイグレーション前など重要な変更の前に取る
```

## セクション4: 破壊的マイグレーションの2段階デプロイ

### なぜ一度にやらないか
- ローリングデプロイ中は新旧ECSタスクが同時に動く
- カラム削除を一気にやると旧タスクがエラーになる

### 2段階の流れ
```
第1段階:
  コードから旧カラムの参照を外し新カラムを使う変更をデプロイ
  → DBにはまだ旧カラムが残っている
  → 新旧どちらのコードでも動く

第2段階（次のリリース）:
  旧コードがもう動いていないことを確認
  → 旧カラムを削除するマイグレーションを実行
  → 安全に削除できる
```

### ポイント
- 無停止デプロイをする全てのシステムに共通する考え方
- CLAUDE.mdにも「破壊的マイグレーションは2段階デプロイで行う」と記載

## セクション5: RDSのネットワーク配置

### プライベートサブネットに配置
```
RDS → プライベートサブネット
  → インターネットから直接アクセス不可
  → ECSからのみ接続可能（SGで制限）
```

### DBサブネットグループ
- RDSを複数のAZに配置するための設定
- AZ 1aとAZ 1cの両方のサブネットを指定
- 障害時にAWSが自動でフェイルオーバー

### 実体験
- ECSコンテナからDBにアクセスできたのは、同じVPC内のプライベートサブネットにいて、SGで通信が許可されているから
