# Day 1: AWSの全体像 - リージョン・AZ・VPC・サブネット

- 学習日: 2026-04-05
- 所要時間: 約30分
- きっかけ: POC環境の学習データアップロードで504エラー → ALB/ECS/S3の調査を通じてインフラの理解が必要と実感

## セクション1: リージョンとAZ

### 学んだこと
- **リージョン** = AWSのデータセンター群がある地域（ap-northeast-1 = 東京）
- **AZ（アベイラビリティゾーン）** = リージョン内の独立したデータセンター
- 東京リージョンには3つのAZ: 1a, 1c, 1d

### 確認コマンド
```bash
aws ec2 describe-availability-zones --region ap-northeast-1 --query "AvailabilityZones[*].{ZoneName:ZoneName,State:State}" --output table
```

### ポイント
- DandoriAIが2つのサブネットを持つ理由 = 異なるAZに配置して耐障害性を確保
- AZ 1aが障害 → 1cで継続。逆も然り

---

## セクション2: VPC

### 学んだこと
- **VPC** = AWS上に作る自分専用の仮想ネットワーク（仮想世界）
- IPアドレスの範囲（CIDR）を割り当てる
- DandoriAIは2つのVPCを持つ:
  - 本番: `10.0.0.0/16`
  - POC: `10.1.0.0/16`
- VPC同士は完全に分離されている（本番からPOCのDBに直接アクセス不可）

### 確認コマンド
```bash
aws ec2 describe-vpcs --query "Vpcs[*].{VpcId:VpcId,CIDR:CidrBlock,Name:Tags[?Key=='Name'].Value|[0]}" --output table
```

### つまづいたところ
- CIDRの`/16`の意味が最初分からなかった
- 「先頭16ビットを固定する」= 残り16ビット（65,536個）が自由に使える範囲
- IPアドレスは32ビット = 4バイト。`10.0.0.0`の各数字(0-255)は8ビット

## セクション3: サブネット

### 学んだこと
- **サブネット** = VPCの中を区切った小部屋（マンションの各フロア）
- VPCが「仮想世界」、AZが「マンション」、サブネットが「フロア」

### DandoriAI POC環境のサブネット構成
```
VPC（10.1.0.0/16）= 仮想世界

マンションA（AZ 1a）
  1階 public-1（10.1.0.0/24）← ALBがいる
  2階 private-1（10.1.10.0/24）← ECS/RDSがいる

マンションB（AZ 1c）
  1階 public-2（10.1.1.0/24）← ALBがいる
  2階 private-2（10.1.11.0/24）← ECS/RDSがいる
```

### パブリック vs プライベート
- **パブリックサブネット**: インターネットゲートウェイへのルートがある → 外からアクセス可能
- **プライベートサブネット**: そのルートがない → 外から直接アクセス不可
- サブネット自体にラベルはなく、**ルートテーブル**の設定で決まる
- プライベートから外への通信は**NATゲートウェイ**経由（一方通行）

### つまづいたところ
- 「パブリック/プライベートの違いは何で決まるのか」が分からなかった
- → ルートテーブルにインターネットゲートウェイへのルートがあるかどうか

## セクション4: リクエストの流れ

### 全体像
```
ブラウザ
  → DNS解決（Route53がALBのIPを返す）
    → インターネットゲートウェイ
      → ALB（パブリックサブネット）
        → ECSコンテナ（プライベートサブネット）
          → レスポンスを逆に辿って返す
```

### 実体験との対応
| 今日やったこと | 対応する場所 |
|---|---|
| `dig`でIP確認 | DNS解決 |
| ALBのidle_timeout変更 | ALB |
| 502/504エラー | ALB → ECSの間で問題 |
| OOM Kill、ECSメモリ増量 | ECSコンテナ |
| S3アップロード | ECSから別サービスへ通信 |
